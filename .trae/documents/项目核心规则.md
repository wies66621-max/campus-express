## 一、项目最高目标

### 🎯 核心原则
本项目以**"通过毕业设计答辩"**为最高目标，而非工程复杂度或技术炫技。

### ✅ 所有智能体必须遵守

- ❌ 不引入课程未系统讲解或明显偏工程化的技术
- ❌ 不追求"企业级""高并发""分布式""微服务"
- ✅ 以 **功能完整、逻辑清晰、易于讲解** 为第一标准
- ✅ 代码必须 **适合老师现场查看、学生本人解释**

### 📋 开发标准
1. **代码可读性** > 代码简洁性
2. **功能完整性** > 技术先进性
3. **易于讲解** > 架构完美
4. **老师能看懂** > 同行能看懂

## 三、功能复杂度限制规则

### 1️⃣ 业务逻辑限制

#### 快递状态管理
**✅ 允许**:
- 使用枚举 + 简单状态流转
- 状态：pending（待取件）→ picked（已取件）→ expired（超期）

**❌ 禁止**:
- 状态机框架
- 复杂状态流转逻辑

**示例代码**:
```java
public enum ExpressStatus {
    PENDING("待取件"),
    PICKED("已取件"),
    EXPIRED("超期");

    private final String desc;

    ExpressStatus(String desc) {
        this.desc = desc;
    }
}

// 简单状态流转
public void pickupExpress(Long expressId) {
    Express express = getById(expressId);
    if (!ExpressStatus.PENDING.equals(express.getStatus())) {
        throw new BusinessException("快递状态不允许取件");
    }
    express.setStatus(ExpressStatus.PICKED);
    updateById(express);
}
```

---

#### 权限控制
**✅ 允许**:
- 基于角色（学生 / 快递员 / 管理员）
- 简单的@RequireAuth注解

**❌ 禁止**:
- 细粒度RBAC / 权限树
- 复杂的权限控制框架

**示例代码**:
```java
// 用户角色枚举
public enum UserRole {
    USER("普通用户"),
    ADMIN("管理员"),
    COURIER("快递员");

    private final String desc;

    UserRole(String desc) {
        this.desc = desc;
    }
}

// 简单权限注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireAuth {
    UserRole[] value() default {};
}

// 使用示例
@RequireAuth(UserRole.ADMIN)
public void deleteExpress(Long id) {
    removeById(id);
}
```

#### 认证机制
**✅ 允许**:
- JWT + 拦截器
- 简单的token验证

**❌ 禁止**:
- 刷新Token
- 双Token
- 黑名单等复杂机制

**示例代码**:
```java
// JWT工具类（简化版）
public class JwtUtil {
    private static final String SECRET = "campus-express-secret";
    private static final long EXPIRATION = 86400000; // 24小时

    public static String generateToken(Long userId, String username) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", userId);
        claims.put("username", username);
        return Jwts.builder()
                .setClaims(claims)
                .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
                .signWith(SignatureAlgorithm.HS256, SECRET)
                .compact();
    }

    public static Claims parseToken(String token) {
        return Jwts.parser()
                .setSigningKey(SECRET)
                .parseClaimsJws(token)
                .getBody();
    }
}

// JWT拦截器（简化版）
public class JwtInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        String token = request.getHeader("Authorization");
        if (token == null) {
            throw new BusinessException("未登录");
        }
        Claims claims = JwtUtil.parseToken(token);
        request.setAttribute("userId", claims.get("userId"));
        return true;
    }
}
```

### 2️⃣ 数据库设计限制

#### 表数量建议
- **建议**: 6～10张表
- **本项目**: 4张表（user、express、pickup_record、station）

#### 禁止
- ❌ 多对多中间表复杂嵌套
- ❌ 触发器、存储过程

#### 允许
- ✅ 单表CRUD
- ✅ 简单外键逻辑（可逻辑不物理）

**示例代码**:
```sql
-- 简单的外键关系（逻辑关联）
CREATE TABLE express (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  express_no VARCHAR(50) NOT NULL UNIQUE,
  recipient_name VARCHAR(50) NOT NULL,
  recipient_phone VARCHAR(20) NOT NULL,
  pickup_code VARCHAR(10) NOT NULL UNIQUE,
  status VARCHAR(20) DEFAULT 'pending',
  storage_time DATETIME NOT NULL,
  pickup_time DATETIME,
  station_id BIGINT COMMENT '快递站ID（逻辑外键）',
  admin_id BIGINT COMMENT '操作管理员ID（逻辑外键）',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) COMMENT='快递表';

-- 简单的查询关联（不使用物理外键）
public List<Express> listByStationId(Long stationId) {
    return expressMapper.selectList(
        new LambdaQueryWrapper<Express>()
            .eq(Express::getStationId, stationId)
    );
}
```

## 四、代码风格强制要求

### ✅ 所有智能体生成代码时必须遵守

#### 1. 命名规范
- ✅ 类名、方法名语义清晰
- ✅ 使用驼峰命名法
- ✅ 常量使用大写下划线

**示例**:
```java
// ✅ 正确
public class ExpressService {
    public Express getExpressById(Long id) {
        return expressMapper.selectById(id);
    }

    public String generatePickupCode() {
        return RandomUtil.randomString("0123456789", 6);
    }
}

// ❌ 错误
public class ES {
    public Express get(Long i) {
        return mapper.selectById(i);
    }
}
```

---

#### 2. 注释要求
- ✅ 每个Controller方法有中文注释
- ✅ 复杂业务逻辑有注释
- ✅ 关键算法有注释

**示例**:
```java
@Api(tags = "快递管理")
@RestController
@RequestMapping("/express")
public class ExpressController {

    @Autowired
    private ExpressService expressService;

    /**
     * 快递入库
     * 功能：管理员录入快递信息，系统自动生成取件码
     * 参数：快递单号、收件人姓名、收件人手机号、快递站ID
     * 返回：6位取件码
     */
    @ApiOperation("快递入库")
    @PostMapping("/storage")
    public R<String> storage(@Valid @RequestBody ExpressDTO dto) {
        String pickupCode = expressService.saveExpress(dto);
        return R.success(pickupCode);
    }

    /**
     * 快递取件
     * 功能：用户通过取件码取件，系统更新快递状态
     * 参数：取件码
     * 返回：取件成功信息
     */
    @ApiOperation("快递取件")
    @PostMapping("/pickup")
    public R<String> pickup(@RequestParam String pickupCode) {
        expressService.pickupExpress(pickupCode);
        return R.success("取件成功");
    }
}
```

#### 4. 代码复杂度
- ❌ 不生成"高度抽象工具类"
- ❌ 不省略关键业务代码

**原则**:
> 宁可代码"啰嗦一点"，也不要"看不懂"。

**示例**:
```java
// ✅ 正确（清晰易懂）
public String generatePickupCode() {
    String pickupCode;
    int maxAttempts = 10;
    int attempts = 0;

    do {
        pickupCode = RandomUtil.randomString("0123456789", 6);
        attempts++;
        if (attempts >= maxAttempts) {
            throw new BusinessException("取件码生成失败，请重试");
        }
    } while (expressMapper.existsByPickupCode(pickupCode));

    return pickupCode;
}

// ❌ 错误（过度抽象）
public String generateUniqueCode(String charset, int length, int maxAttempts, Function<String, Boolean> existsChecker) {
    return Stream.generate(() -> RandomUtil.randomString(charset, length))
            .filter(code -> !existsChecker.apply(code))
            .limit(maxAttempts)
            .findFirst()
            .orElseThrow(() -> new BusinessException("生成失败"));
}
```

## 五、文档 & 论文章节适配规则

### 📝 技术选型三问

所有技术，必须能回答这3个问题：

1. **为什么要用它？**
2. **不用它会有什么问题？**
3. **它在本系统中具体解决了什么？**

### ⚠️ 技术使用原则

如果某个技术**回答不了这3个问题**：
- 👉 智能体**不得使用**

### 📋 技术选型示例

#### 示例1：MyBatis-Plus
**为什么要用它？**
- 简化数据库操作，减少重复代码
- 提供分页、条件构造等常用功能

**不用它会有什么问题？**
- 需要编写大量重复的SQL语句
- 分页、条件查询等需要自己实现
- 开发效率低，代码冗余

**它在本系统中具体解决了什么？**
- 快速实现快递、用户等模块的CRUD操作
- 实现快递列表的分页查询
- 实现按状态、收件人等条件的筛选查询

---

#### 示例2：JWT
**为什么要用它？**
- 无状态的身份认证机制
- 适合前后端分离架构

**不用它会有什么问题？**
- 需要使用Session，不适合前后端分离
- 需要服务器存储会话信息
- 扩展性差

**它在本系统中具体解决了什么？**
- 实现用户登录后的身份认证
- 保护需要登录才能访问的接口
- 前端通过token识别用户身份

---

#### 示例3：Knife4j
**为什么要用它？**
- 自动生成API文档
- 提供在线测试接口功能

**不用它会有什么问题？**
- 需要手动编写API文档
- 文档更新不及时
- 接口测试不方便

**它在本系统中具体解决了什么？**
- 自动生成快递管理、用户管理等模块的接口文档
- 方便老师查看和测试接口
- 便于答辩时演示接口功能

## 七、常见问题解答

### Q1: 为什么不使用Redis？
**A**: Redis是缓存技术，虽然能提升性能，但：
- 课程未系统讲解
- 增加系统复杂度
- 对于毕设项目，MySQL已足够
- 答辩时难以解释清楚

### Q2: 为什么不使用微服务？
**A**: 微服务架构虽然先进，但：
- 超出毕设要求范围
- 增加开发和部署复杂度
- 单体应用更适合毕设演示
- 答辩时难以讲清楚

### Q3: 为什么不使用Pinia？
**A**: Pinia是状态管理库，但：
- 项目规模小，不需要复杂状态管理
- 增加学习成本
- 原生Vue响应式已足够
- 答辩时难以解释其必要性

### Q4: 为什么代码要"啰嗦"一点？
**A**: 因为：
- 毕设答辩时，老师需要看懂代码
- 复杂的抽象会增加理解难度
- 清晰的代码更容易讲解
- 宁可代码长一点，也不要看不懂
