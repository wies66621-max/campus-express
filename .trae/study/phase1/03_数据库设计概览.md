## 📊 数据库表结构

### 表1：express（快递表）

**这是什么表？**
- 存储所有快递的信息
- 包括待取件和已取件的快递

**主要字段：**

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| id | Long | 快递ID（主键） | 1, 2, 3... |
| tracking_number | String | 快递单号 | SF1234567890 |
| company | String | 快递公司 | 顺丰、圆通、中通 |
| receiver_name | String | 收件人姓名 | 张三 |
| receiver_phone | String | 收件人手机号 | 13800138000 |
| pickup_code | String | 取件码 | 123456 |
| status | Integer | 状态（0=待取件，1=已取件） | 0或1 |
| station_id | Long | 快递站ID | 1, 2, 3... |
| remark | String | 备注 | 易碎品 |
| create_time | LocalDateTime | 创建时间 | 2024-01-01 10:00:00 |
| update_time | LocalDateTime | 更新时间 | 2024-01-01 12:00:00 |

**生活化例子：**
就像快递单，上面写着快递单号、收件人姓名、收件人电话、取件码、是否已取件等信息。

**为什么要这样设计？**
- `tracking_number`是唯一的，用来区分不同的快递
- `pickup_code`是6位随机数，方便用户取件
- `status`标记快递是否已取件
- `station_id`关联快递站，知道快递在哪个站点

**如果删掉这个表会怎样？**
- 无法存储快递信息
- 整个系统无法使用

---

### 表2：user（用户表）

**这是什么表？**
- 存储所有用户的信息
- 包括管理员、快递员、学生

**主要字段：**

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| id | Long | 用户ID（主键） | 1, 2, 3... |
| username | String | 用户名（登录用） | admin, student1 |
| password | String | 密码（加密存储） | 加密后的字符串 |
| real_name | String | 真实姓名 | 张三 |
| phone | String | 手机号 | 13800138000 |
| role | String | 角色（admin/courier/user） | admin, courier, user |
| create_time | LocalDateTime | 创建时间 | 2024-01-01 10:00:00 |
| update_time | LocalDateTime | 更新时间 | 2024-01-01 12:00:00 |

**生活化例子：**
就像会员卡，上面写着会员号、姓名、手机号、会员等级等信息。

**为什么要这样设计？**
- `username`和`password`用于登录
- `role`区分不同角色的权限
- `phone`用于快递收件人匹配

**如果删掉这个表会怎样？**
- 用户无法登录
- 无法区分不同角色的权限
- 无法匹配快递收件人

---

就像快递站的招牌，上面写着快递站名称、地址、负责人、联系电话等信息。

**为什么要这样设计？**

### 表4：pickup_record（取件记录表）

**这是什么表？**
- 存储所有取件记录
- 记录谁在什么时候取了哪个快递

**主要字段：**

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| id | Long | 记录ID（主键）） | 1, 2, 3... |
| express_id | Long | 快递ID | 1, 2, 3... |
| operator_id | Long | 操作员ID（谁取的件） | 1, 2, 3... |
| pickup_time | LocalDateTime | 取件时间 | 2024-01-01 12:00:00 |
| status | Integer | 状态（0=失败，1=成功） | 0或1 |
| remark | String | 备注 | 取件成功 |
| create_time | LocalDateTime | 创建时间 | 2024-01-01 12:00:00 |

**生活化例子：**
就像取件登记本，上面写着谁在什么时候取了哪个快递，取件是否成功。

**为什么要这样设计？**
- `express_id`关联快递，知道取的是哪个快递
- `operator_id`关联操作员，知道是谁取的件
- `pickup_time`记录取件时间
- `status`标记取件是否成功

**如果删掉这个表会怎样？**
- 无法追溯取件记录
- 无法统计取件数据
- 无法知道谁在什么时候取了件

## 🔗 表之间的关系

### 1. express（快递表）和 user（用户表）的关系

**关系类型：多对一**

**解释：**
- 一个用户可以有多个快递
- 一个快递只能有一个收件人

**关联字段：**
- express.receiver_phone = user.phone

**生活化例子：**
就像学生和快递的关系，一个学生可以收多个快递，但一个快递只能有一个收件人。

---

### 2. express（快递表）和 station（快递站表）的关系

**关系类型：多对一**

**解释：**
- 一个快递站可以有多个快递
- 一个快递只能在一个快递站

**关联字段：**
- express.station_id = station.id

**生活化例子：**
就像快递和快递站的关系，一个快递站可以存放多个快递，但一个快递只能在一个快递站。

---

### 3. express（快递表）和 pickup_record（取件记录表）的关系

**关系类型：一对多**

**解释：**
- 一个快递可以有多条取件记录（理论上不应该，但设计上允许）
- 一条取件记录只能对应一个快递

**关联字段：**
- pickup_record.express_id = express.id

**生活化例子：**
就像快递和取件记录的关系，一个快递可能被取多次（虽然实际不会），但每次取件都有一条记录。

---

### 4. pickup_record（取件记录表）和 user（用户表）的关系

**关系类型：多对一**

**解释：**
- 一个用户可以有多条取件记录
- 一条取件记录只能对应一个操作员

**关联字段：**
- pickup_record.operator_id = user.id

**生活化例子：**
就像操作员和取件记录的关系，一个操作员可以操作多次取件，但每次取件只能有一个操作员。

---

### 5. notice（公告表）和 user（用户表）的关系

**关系类型：多对一**

**解释：**
- 一个用户可以发布多条公告
- 一条公告只能由一个用户发布

**关联字段：**
- notice.creator_id = user.id

**生活化例子：**
就像用户和公告的关系，一个用户可以发布多条公告，但一条公告只能由一个用户发布。

## 📐 E-R图（实体关系图）

### 文字版E-R图

```
┌─────────────┐
│   user      │
│  (用户表)   │
└──────┬──────┘
       │ 1
       │
       │ N
┌──────▼──────┐         ┌─────────────┐
│   express   │         │   station   │
│  (快递表)   │────────▶│ (快递站表)  │
└──────┬──────┘         └─────────────┘
       │ 1
       │
       │ N
┌──────▼──────────────┐
│  pickup_record      │
│   (取件记录表)       │
└─────────────────────┘

┌─────────────┐
│   notice    │
│  (公告表)   │
└──────┬──────┘
       │ 1
       │
       │ N
┌──────▼──────┐
│   user      │
│  (用户表)   │
└─────────────┘
```

### E-R图说明

- **矩形**：表示实体（表）
- **线条**：表示关系
- **1**：表示"一"端
- **N**：表示"多"端

**关系解释：**
1. user和express：1对N（一个用户可以有多个快递）
2. express和station：N对1（多个快递属于一个快递站）
3. express和pickup_record：1对N（一个快递可以有多条取件记录）
4. pickup_record和user：N对1（多条取件记录属于一个操作员）
5. notice和user：N对1（多条公告属于一个发布人）

## 🎯 整体观：数据库设计在项目中的位置

### 数据库在系统中的作用

```
用户
  ↓ 前端页面
  ↓ HTTP请求
后端Controller
  ↓ 调用
后端Service
  ↓ 调用
后端Mapper
  ↓ SQL操作
数据库（存储数据）
```

### 数据流

1. **快递入库流程**
   - 管理员在前端输入快递信息
   - 前端发送HTTP请求到后端
   - 后端Service生成取件码
   - 后端Mapper插入数据到express表
   - 数据库保存快递信息

2. **快递取件流程**
   - 用户在前端输入取件码
   - 前端发送HTTP请求到后端
   - 后端Service验证取件码
   - 后端Mapper更新express表的status字段
   - 后端Mapper插入数据到pickup_record表
   - 数据库保存取件记录

3. **快递查询流程**
   - 用户在前端输入查询条件
   - 前端发送HTTP请求到后端
   - 后端Service构建查询条件
   - 后端Mapper查询express表
   - 数据库返回查询结果

## 📌 答辩可用说法示例

### 老师如果问"你的数据库有哪些表？"

**你可以这样说：**

> "我的数据库一共有5张表：第一张是express（快递表），存储所有快递的信息，包括快递单号、收件人、取件码、状态等；第二张是user（用户表），存储所有用户的信息，包括用户名、密码、角色等；第三张是station（快递站表），存储所有快递站的信息，包括快递站名称、地址、负责人等；第四张是pickup_record（取件记录表），存储所有取件记录，包括取件时间、操作员等；第五张是notice（公告表），存储所有公告信息，包括公告标题、内容等。"

### 老师如果问"表和表之间有什么关系？"

**你可以这样说：**

> "表和表之间主要有以下几种关系：第一，express表和user表是多对一关系，一个用户可以有多个快递，但一个快递只能有一个收件人；第二，express表和station表是多对一关系，一个快递站可以有多个快递，但一个快递只能在一个快递站；第三，express表和pickup_record表是一对多关系，一个快递可以有多条取件记录；第四，pickup_record表和user表是多对一关系，一个操作员可以有多条取件记录；第五，notice表和user表是多对一关系，一个用户可以发布多条公告。"

### 老师如果问"你为什么这样设计数据库？"

**你可以这样说：**

> "我的数据库设计遵循了数据库设计的三大范式：第一范式确保每个字段都是不可分割的；第二范式确保非主键字段完全依赖于主键；第三范式确保非主键字段不传递依赖于主键。这样的设计可以减少数据冗余，避免数据不一致，提高数据完整性。同时，我根据业务需求设计了表之间的关系，比如快递和快递站是多对一关系，因为一个快递站可以存放多个快递，但一个快递只能在一个快递站。"

### 老师如果问"如果让你重新设计数据库，你会怎么改进？"

**你可以这样说：**

> "如果重新设计，我可能会考虑以下几点：第一，增加软删除字段，删除数据时不真正删除，而是标记为已删除，这样可以保留历史记录；第二，增加索引，提高查询性能，比如在tracking_number、pickup_code、receiver_phone等字段上建立索引；第三，增加数据字典表，统一管理状态码、角色码等，避免硬编码；第四，增加日志表，记录关键操作，便于问题追溯。不过，对于本科毕设来说，当前的数据库设计已经足够了。"

### 老师如果追问"什么是数据库三大范式？"

**你可以这样说：**

> "数据库三大范式是数据库设计的规范：第一范式要求每个字段都是不可分割的原子值，比如不能把姓名和电话放在一个字段里；第二范式要求非主键字段完全依赖于主键，不能只依赖于主键的一部分，这主要针对联合主键；第三范式要求非主键字段不传递依赖于主键，不能通过其他非主键字段间接依赖于主键。遵循三大范式可以减少数据冗余，避免数据不一致，提高数据完整性。"
